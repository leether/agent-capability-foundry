# Agent Capability Foundry 设计理念

状态：派生文档

## 用途

这份文档定义未来公开仓库 `agent-capability-foundry` 的治理理念。

它不是一份风格说明。

它是一个仓库级约束系统，用来约束：

- 什么内容可以进入仓库
- 内容应如何组织
- 矛盾出现时如何处理
- 仓库如何在不失去整体完整性的前提下演化

## 核心要求

这组治理约束必须能够自指。

也就是说，这组约束不仅要能评估仓库内容，还要能评估它自身。

如果这些原则不能作用到它们自己的表述、边界和修订方式上，那么它们就不足以治理一个活的方法论仓库。

## 治理四元组

这个仓库由四个核心约束共同治理：

1. 边界
2. 完备性
3. 自洽性
4. 自进化

这四个约束必须协同工作。

不能把它们当成彼此独立的口号。

## 边界

每一条重要规则、每一个 capability、每一份文档都必须说明：

- 它治理什么范围
- 它不治理什么范围
- 它作用在哪个时间尺度上
- 需要什么证据才可以修订

边界的作用是防止出现“伪完备性”。

没有边界，完备性就会无限膨胀，最终无法使用。

## 完备性

完备性的含义是：

- 在既定边界内，仓库应覆盖足以让其所宣称的方法论可操作的主要层

对这个仓库来说，这些主要层包括：

- capability 定义
- workflow / 编排
- tool use
- safety / approvals
- eval / validation
- context / knowledge / memory
- runtime portability

完备性并不意味着“什么都要有”。

它的意思是：

- 如果缺失了让方法论真正可操作的关键层，就不能把它宣称为一个完整的方法论

## 自洽性

自洽性的含义是：

- 在某一个既定发布态中，仓库的各层之间不能互相打架

以下这些层不应互相矛盾：

- 仓库名称
- 顶层抽象
- 文件布局
- 安装入口
- runtime adapter 模型
- 文档
- 双语机制
- 收录规则

自洽性的作用是防止隐藏例外和用户困惑。

## 自进化

自进化的含义是：

- 当仓库遇到认知盲点、内部矛盾或新模式时，必须能够显式修订自己，而不是僵死不变，或在无声漂移中失控

自进化并不是在反对完备性和自洽性。

它治理的是“状态之间如何变化”。

完备性和自洽性衡量的是某个状态本身的质量。
自进化衡量的是从旧状态走向新状态的变化质量。

## 四元组如何协同

- 边界约束“什么范围内需要完备”
- 完备性约束“在这个范围内不能漏掉哪些关键层”
- 自洽性约束“这些层在同一时刻如何不互相打架”
- 自进化约束“当新证据挑战当前结构时，仓库应如何变化”

这样四元组就能形成一种可递归但稳定的结构：

- 一个发布态应当有清楚边界、足够完备、并保持自洽
- 一次修订应当显式、基于证据，并且能够回到新的自洽状态

## 自指测试

这四元组也必须能作用于它自己。

### 四元组的边界

这四元组治理的是：

- 仓库设计
- 收录规则
- 安装模型
- runtime portability
- 文档体系
- 修订规则

它并不宣称自己是适用于所有组织或所有知识形态的普适理论。

### 四元组的完备性

对“仓库治理”这个问题来说，这个四元组已经足够完备，因为它覆盖了：

- 范围控制
- 当前状态质量
- 跨层一致性
- 结构化修订

但它并不是对所有哲学问题都绝对完备。

### 四元组的自洽性

这个四元组只有在按不同时间尺度理解时才是自洽的：

- 边界、完备性、自洽性治理的是发布态
- 自进化治理的是状态之间的修订

如果把自进化理解成“允许长期不自洽”，那这个四元组就会失去自洽性。

### 四元组的自进化

这个四元组本身也允许被修订。

但修订必须满足：

- 显式
- 有版本
- 有理由
- 有证据
- 修订后回到新的自洽状态

四元组本身不能通过无声语义漂移来进化。

## 矛盾处理原则

矛盾不自动等于失败。

很多时候，矛盾说明的是下面几类问题之一：

- 当前覆盖面存在盲点
- 各层之间出现错配
- 合约已经过时
- 出现了当前模型还无法表达的新有效模式

仓库应当这样响应：

1. 识别矛盾
2. 局部化矛盾
3. 给矛盾命名
4. 判断它属于哪一类：
   - 边界调整
   - 完备性扩展
   - 自洽性修复
   - 显式演化
5. 修订受影响的层
6. 恢复新的自洽发布态

## 操作规则

仓库应当通过以下方式演化：

- 显式设计文档
- 带版本的 capability contract
- maturity states
- deprecation paths
- 在进入 curated surface 之前先做验证

仓库不应通过以下方式演化：

- 未记录的例外
- 在规范性文档中长期保留互相矛盾的真相
- 隐式改变规范术语的含义
- 把实验性内容伪装成稳定 canon

## 设计后果

这套理念意味着：

- 仓库追求的是“结构化进化”，不是“静态僵硬”
- 在探索阶段可以出现矛盾
- 但在 curated canonical surface 中，矛盾不能长期不解决
- 单点好用并不足够；如果它削弱了仓库级完整性，就不能直接进入规范层

## 规范性结论

这个仓库应始终保持：

- 有边界意识
- 在其声明范围内足够完备
- 在每一个已发布状态中保持自洽
- 当新证据出现时，通过显式自进化完成修订

这就是 `agent-capability-foundry` 的治理理念。
